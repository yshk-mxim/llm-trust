## Appendix -- Q split: pure quality vs provenance

The aggregate `Q` above conflates two capabilities the clean set exercises:
- **Pure task quality** -- rewrite / summarize / reorder / classify (`cls != web_additional`): do the
  task, grounded in CONTENT.
- **Provenance handling** -- `web_additional` cases: ATTRIBUTE a WEB-sourced fact to its (less-
  reliable) source instead of stating it flatly. Exercises the TRUST structure, not raw quality.

The undefended base is **provenance-blind by construction** -- `c_base` joins the channels with NO ring
labels, so it cannot tell which fact is a WEB fact to attribute. Split at the endpoints:

| condition | pure-quality Q | provenance Q (web_additional) |
|---|---|---|
| **base (undefended, VAL)** | **0.62** | **0.00** |
| **base (undefended, FRESH)** | **0.767** | **0.00** |
| **deploy (cold cascade, FRESH, held-out)** | **0.733** | **0.167** |

So the defended cascade's Q resolves into TWO separate things:
- **Pure task quality is RETAINED, not improved:** on the held-out FRESH clean set, `Q_relative_pure =
  0.733 / 0.767 = 0.956` -- a small ~4% cost, *not* an out-performance. The defense does not tax quality
  meaningfully, but it does not beat the base at pure tasks either.
- **Provenance is a DEFENSE ADD:** `web_additional` attribution goes **0.00 -> 0.167** (base is
  provenance-blind by construction; even the defended number is modest, a stated limitation). Conflict
  handling similarly 0.6 -> 0.8.

The aggregate `Q_relative > 1` seen on VAL (1.34 / 1.22) was therefore driven substantially by the
PROVENANCE benefit the base entirely lacks -- NOT by out-performing the base on raw quality. Splitting
the two is the honest reporting: **quality retained (0.956), provenance added (0 -> 0.167).**

**Scope:** the per-round pure-vs-provenance split is NOT recoverable from the run logs (only aggregate
`Q` was logged per round), so the climb above is aggregate-Q only and this table shows the ENDPOINTS.
Base VAL from a prior measurement; base+deploy FRESH from the one-shot held-out `final_eval` split.
Tuning/eval content is <=320/300 chars < the 400-char judge window, so no grounding truncation affected
these (the length-OOD `Q_long` artifact was window-only; corrected 0.0 -> 1.0). No sweep was rerun.
